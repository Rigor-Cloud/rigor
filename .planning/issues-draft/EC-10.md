# EC-10: Proxy cutover — replace `build_epistemic_context` with `EpistemicCortex::assemble_for_request`

> Part of umbrella: #34 [UMBRELLA] Epistemic Cortex
> Depends on: **EC-1** through **EC-9**
> Lands in: `crates/rigor/src/memory/epistemic/cortex.rs`, `crates/rigor/src/daemon/proxy.rs`, config wiring

## Scope

The first user-visible change. After this lands:

- `src/daemon/proxy.rs:1256` stops calling `build_epistemic_context(&st.config, &st.graph)` and starts calling `st.cortex.assemble_for_request(&req).await?`.
- `extract_and_evaluate_text` in proxy.rs is extended to call `st.cortex.record_response(session_id, claims, verdicts).await` at the end, so the cortex learns from each turn.
- The swap is behind a feature flag `epistemic_cortex` in rigor.yaml. Default is `false` initially; can be flipped on per-project.
- When flag is OFF: 100% of behavior matches pre-EC-10 code. No new code paths exercised.
- When flag is ON: `EpistemicCortex::assemble_for_request` drives context injection; working memory and retrieval are active; verification loop ticks every N requests.
- Rigor.yaml's `epistemic:` section is fully wired: embedder, decay, retrieval, verification knobs all propagate into the cortex's configured behavior.
- The daemon schedules `cortex.tick()` every `pass_interval_requests` proxy requests to drive the verification loop.

## Design constraints pinned from the design thread

- **Flag-gated rollout.** First version ships with `epistemic_cortex: false` by default. Developers enable per-project; once validated, defaults flip to true.
- **Zero regression when flag OFF.** The cortex code paths are fully inert when disabled. No DB writes, no embedder loads, no background ticks.
- **One-line diff at `proxy.rs:1256`.** That's the cutover site; everything else is trait wiring.
- **`record_response` at the end of `extract_and_evaluate_text`.** Closed loop: rigor writes what it observed; the next turn's retrieval sees it.
- **Verification loop is turn-driven.** Every `pass_interval_requests` resolves from `SessionResolver::resolve`, the daemon calls `cortex.tick(budget)`.
- **Config-driven half-lives and thresholds.** The cortex's behavior is a function of `rigor.yaml` + event state. No hardcoded policies.

    store:        Arc<dyn EpistemicStore>,
    wm:           Arc<dyn WorkingMemory>,
    retrieval:    Arc<dyn RetrievalEngine>,
    inhibition:   Arc<dyn InhibitionLedger>,

impl EpistemicCortex {
    /// Build an EpistemicCortex from config + substrate.
    /// Returns None if `epistemic_cortex` flag is disabled.
    pub async fn new_from_config(cfg: &EpistemicConfig, substrate: Arc<SqliteSubstrate>) -> Result<Option<Self>>;

        claims: &[Claim],

    pub async fn hook_session_start(&self, hook_id: &str, agent: AgentKind) -> Result<()>;
    pub async fn hook_session_end(&self, hook_id: &str) -> Result<()>;

    /// Called periodically by the daemon's tick loop.
    /// Returns PassReport for observability.
    pub async fn tick(&self, budget: VerificationBudget) -> Result<PassReport>;

    /// Invalidate the cached stable preamble — called after rigor.yaml reload.
    pub async fn invalidate_preamble(&self) -> Result<()>;
}

### Daemon tick scheduling

The daemon's existing tick infrastructure gets one addition:

```rust
// In daemon::mod.rs or wherever the tick loop lives
async fn on_request_completed(state: &DaemonState, session_id: &str) {
    let requests_since_last_pass = state.requests_since_last_pass.fetch_add(1, Ordering::Relaxed);
    if requests_since_last_pass >= state.cortex_config.verification.pass_interval_requests {
        state.requests_since_last_pass.store(0, Ordering::Relaxed);
        if let Some(cortex) = &state.cortex {
            let budget = VerificationBudget::from(&state.cortex_config.verification);
            match cortex.tick(budget).await {
                Ok(report) => tracing::info!(verifed=report.beliefs_verified, drifted=report.beliefs_drifted, "cortex tick"),
                Err(e) => tracing::warn!("cortex tick failed: {e:?}"),
            }
        }
    }
}
```

## `record_response` flow (authoritative)

```
record_response(session, claims, verdicts):
    for (claim, verdict) in zip(claims, verdicts):
        belief_id = canonical_id_for_claim(claim)

        # 1. Append BeliefAsserted event
        store.append_event(BeliefAsserted {
            kind: Claim,
            knowledge_type: claim.knowledge_type,
            payload_json: serialize(claim),
            base_strength: default_strength_for(claim.knowledge_type),
            source_id: detect_source(req.model),          // 'claude-opus-4-7' etc.
        })

        # 2. Embed + store embedding (non-blocking)
        embedder.embed(claim.text).then |vec| store.store_embedding(belief_id, vec)

        # 3. Activate in working memory
        role = match verdict.decision:
            Block => Contradicting,
            Warn  => UnderEvaluation,
            Allow => Surfaced,
        wm.activate(session, belief_id, role, initial_activation=0.8)

        # 4. Detect contradictions against active WM
        findings = contradiction.detect(session, claim, verdict)
        for finding in findings:
            store.append_event(Contradicted { by: belief_id, evidence_json: serde(finding) })
            inhibition.inhibit(finding.against_belief_id, ContradictedBy, cause=Contradicted event)

        # 5. Source outcome update
        if verdict.decision == Block OR findings.not_empty():
            sources.on_contradiction(source_id, event)
        else if verdict.decision == Allow AND claim.knowledge_type == Empirical:
            # Implicit "verified by rigor's evaluator" — counts toward validation
            sources.on_validation(source_id, event)
        else:
            sources.on_assertion(source_id, event)

    # 6. Goal extraction if first message of session
    if session.turn_count == 1 and req.first_user_message:
        goals.extract_from_first_message(session, req.first_user_message)
```

## Implementation notes & invariants

**Invariant 1: `cortex.record_response` must never block the response.** It's spawned as a tokio task. If it fails, log a warning; don't crash the proxy path. Users would experience a memory-update failure as a stalled response otherwise.

**Invariant 2: feature flag OFF = byte-identical to pre-EC-10.** Test `epistemic_cortex_flag_off_parity.rs` asserts this via snapshot comparison of upstream request bodies.

**Invariant 3: cache_control markers are inserted correctly.** The Anthropic API expects `cache_control: { type: "ephemeral" }` on specific message content blocks. `assembled_to_body_with_cache_markers` constructs this properly; if the API shape changes, only this function needs to be updated.

**Invariant 4: rigor.yaml reload invalidates preamble.** `cortex.invalidate_preamble()` clears the cached stable preamble. Subsequent `assemble_for_request` calls re-render it. `cortex_config_state.rigor_yaml_sha` updated on reload.

**Invariant 5: session_id flows through the entire request.** `assemble_for_request` resolves the session and returns it via `AssembledContext`. The proxy then passes that session_id to `extract_and_evaluate_text`, which passes it to `record_response`. No second session resolve.

**Invariant 6: errors during `assemble_for_request` fall open.** If any cortex call fails mid-assemble (DB down, embedder hang), the proxy falls back to `build_epistemic_context(&st.config, &st.graph)` with a logged warning. The proxy NEVER returns 500 because of cortex problems.

**Invariant 7: `tick` is non-blocking relative to request processing.** The tick is scheduled onto a dedicated tokio task; it doesn't run inline with request handling.

**Operational detail: monitoring.** The cortex emits tracing spans with attributes for mode, retrieved_count, wm_top_k, tokens_used, retrieval_id. These feed the existing observability pipeline (OTel in `src/observability/`).

**Operational detail: verification budget from config.** `VerificationBudget::from(&VerificationConfig)` constructs from rigor.yaml at each tick. Config can be hot-reloaded without daemon restart.

**Operational detail: graceful flag-on migration for existing projects.** When a user sets `epistemic_cortex: true` for the first time:
1. Cortex lazily populates from ongoing activity. No backfill.
2. Cortex is usable immediately but retrieval is initially empty → Empty mode → "novel territory" note for every turn.
3. Over dozens of turns, WM and belief_state_current fill in; retrieval quality ramps up.
4. A separate `rigor epistemic bootstrap` command (not in EC-10) can ingest historical `violations.jsonl` to seed. EC-12 handles the MemoryStore migration; a similar tool for violation-log backfill is follow-on work.
