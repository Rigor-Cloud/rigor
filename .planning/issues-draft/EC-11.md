# EC-11: `rigor refine --gaps` — surface persistent empty-retrieval patterns

> Part of umbrella: #34 [UMBRELLA] Epistemic Cortex
> Depends on: **EC-10**
> Lands in: `crates/rigor/src/cli/refine.rs` extensions; `crates/rigor/src/memory/epistemic/gaps.rs`

## Scope

The feedback loop that closes rigor's "learn-from-usage" story. After this lands:

- `rigor refine --gaps` is a CLI subcommand that queries `retrieval_events` for persistent empty-retrieval patterns.
- Gaps grouped by session goal. A goal with ≥ N (default 3) empty retrievals in the past 30 days is surfaced.
- Each gap suggests remediation: `rigor map --topic X`, or a link to create a manual constraint.
- Output formats: text (default), JSON (`--json`), markdown (`--md`).
- Read-only command; uses the existing reader pool — never blocks the daemon.

## Design constraints pinned from the design thread

- **Empty retrieval is data, not failure.** The presence of many empty retrievals for the same goal signals an epistemic gap worth closing deliberately.
- **Gaps grouped by goal, not query.** Raw query text is too noisy; session goals are the right granularity.
- **Threshold-gated.** A single empty retrieval isn't a gap; persistent gaps are. Default ≥3 in past 30 days; configurable.
- **Suggest remediation.** Don't just report — propose `rigor map --topic X` for common patterns.
- **Read-only CLI.** Uses existing reader pool, doesn't touch the writer lock.

## Trait surfaces

```rust
#[async_trait]
pub trait GapAnalyzer: Send + Sync {
    /// Find persistent gaps over a time window.
    async fn find_gaps(&self, opts: GapQuery) -> Result<Vec<GapCluster>>;
}

pub struct GapQuery {
    pub min_occurrences: usize,        // default 3
    pub lookback_days: i64,             // default 30
    pub limit: usize,                   // default 20 top gaps
    pub min_goal_similarity: f64,       // default 0.75 — how to group similar goals
}

pub struct GapCluster {
    pub representative_goal: String,    // one of the goals in the cluster
    pub cluster_goal_count: usize,      // how many distinct goals folded into this cluster
    pub empty_retrieval_count: usize,
    pub sample_queries: Vec<String>,    // representative queries that returned empty
    pub first_seen: i64,
    pub last_seen: i64,
    pub suggested_remediation: RemediationSuggestion,
}

pub enum RemediationSuggestion {
    RunRigorMap { topic: String, rationale: String },
    AddConstraint { reason: String },
    ExtendKnowledgeGraph { area: String },
    NoObviousRemediation,
}
```

### CLI

```
rigor refine --gaps
    Lists persistent empty-retrieval patterns grouped by session goal.

    Options:
        --lookback DAYS        Time window (default: 30)
        --threshold N          Minimum empty-retrieval count per gap (default: 3)
        --limit N              Top N gaps to display (default: 20)
        --json                 JSON output
        --md                   Markdown output (for pasting into a tracking doc)
        --goal-similarity F    Similarity threshold for goal clustering (default: 0.75)
```

## Implementation notes & invariants

**Invariant 1: goal clustering uses embedding similarity.** Two goals with embeddings cosine similarity ≥ `min_goal_similarity` (default 0.75) are folded into one cluster. Representative goal is the one with the most empty retrievals.

**Invariant 2: read-only execution.** All queries go through the read-only pool. CLI exits with error if rigor.db is missing or not initialized.

**Invariant 3: remediation suggestions are deterministic from clusters.** Rule-based:
- Goal contains domain terms (framework/tool names) → `RunRigorMap { topic }`.
- Goal is about a specific policy decision → `AddConstraint`.
- Goal is about general knowledge → `ExtendKnowledgeGraph`.
- Default → `NoObviousRemediation`.

**Operational detail: retention.** `retrieval_events` grows unbounded. EC-11 queries the last 30 days by default. A future `rigor epistemic gc` command (out of scope) can prune old rows.

## Additional items surfaced in review

- **CLI argument parsing tests.** Use clap-test or the existing rigor CLI test harness to verify:
    - `test_cli_defaults_all_flags_absent` — no flags → all defaults applied.
    - `test_cli_lookback_flag_parsed`.
    - `test_cli_threshold_flag_parsed`.
    - `test_cli_limit_flag_parsed`.
    - `test_cli_json_and_md_mutually_exclusive` — both set → clap error.
    - `test_cli_goal_similarity_out_of_range_rejected` — value outside [0, 1] → error.
- **Exit code semantics.** Document and test:
    - `0` — command succeeded (gaps found OR no gaps found, both normal outcomes).
    - `1` — DB missing / not initialized / incompatible schema version.
    - `2` — CLI argument parse error (handled by clap automatically).
    Add `test_exit_code_0_on_success`, `test_exit_code_1_on_missing_db`.
- **No-gaps output differs from failure.** When no gaps meet threshold, stdout is "No persistent gaps found." (exit 0), distinguishable from "error: database not found" (exit 1, stderr).
- **Deterministic output ordering.** Two consecutive runs against unchanged data produce byte-identical output. Achieved by sorting clusters by `(empty_retrieval_count DESC, representative_goal ASC)`. Add `test_output_deterministic_on_unchanged_data`.
- **Observability (X-1).** `cortex.refine.gaps` span with `lookback_days`, `threshold`, `limit`, `gaps_found`, `query_ms`, `cluster_ms`.
- **No-recursion note.** Read-only CLI; no LLM calls; no X-2 concern.
- **Over-editing as a second gap category (future extension).** The `rigor refine --gaps` analysis surfaces ONE pattern today — goals with persistent empty retrievals. A second pattern worth surfacing, per the 2026 Rehir "Coding Models Are Doing Too Much" article, is **over-editing**: proxy requests where the agent proposed a code change whose normalized token-Levenshtein distance (against the file's pre-change state) or added cognitive-complexity delta materially exceeds what the task required. Concretely:
    - Add a future flag `rigor refine --over-editing`. Or extend the default `--gaps` output with an "Over-editing patterns" section.
    - Metric inputs: (a) token-Levenshtein distance on Python-style tokenized code, normalized to [0, 1]; (b) added cognitive-complexity score (Sonar-style) delta between before and after. These are the metrics from the paper — both are implementable in ~200 lines of Rust.
    - Threshold: edit diffs above the 90th percentile of historical diffs for the same (language, file-type) pair, surfaced when recurring in ≥3 sessions for the same goal.
    - Remediation suggestion: "Add a defeater constraint tagged `edit-minimality` to rigor.yaml for {language}" or "Enable the ContextAssembler preservation instruction (see EC-8 code-editing section)."
    - **Out-of-scope for the EC-11 core landing** — this requires upstream plumbing (Edit/Write tool-use diffs flowing through rigor's claim extraction pipeline, which the current MITM doesn't extract from tool results today). Documented here as the next feature on this CLI surface. A follow-up umbrella (post-EC-12) will pair it with `edit-minimality` constraint family in `src/defaults/<language>.rs`.
