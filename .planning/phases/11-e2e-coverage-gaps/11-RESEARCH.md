# Phase 11: E2E Coverage Gaps - Research

**Researched:** 2026-04-24
**Domain:** End-to-end integration testing of rigor proxy (CONNECT tunneling, TLS MITM, stop-hook, PID lifecycle, corpus replay)
**Confidence:** HIGH

## Summary

Phase 11 closes the E2E gaps listed in REQ-021 that are NOT covered by Phase 12. Phase 12 handles B1/B2/B3 (BLOCK kill-switch, auto-retry, PII-before-upstream). Phase 11 focuses on four distinct test scenarios that exercise foundational proxy and daemon mechanics through real TCP connections:

1. **CONNECT blind-tunnel** -- non-LLM hosts get a byte-for-byte passthrough preserving end-to-end TLS. The proxy issues `tokio::io::copy_bidirectional` between the upgraded connection and the upstream TCP stream.
2. **TLS MITM handshake** -- LLM hosts get a per-host certificate generated by `RigorCA::server_config_for_host()`, signed by the CA. The client must validate this cert against the CA's root certificate.
3. **Stop-hook integration** -- the `run_hook()` function in `lib.rs` reads JSON from stdin, evaluates the constraint pipeline, and emits a JSON response to stdout. This path is tested via subprocess invocation (already partially covered in `integration_hook.rs` and `true_e2e.rs`, but a full through-the-harness test with `IsolatedHome` is missing).
4. **PID file crash recovery** -- `daemon_alive()` must correctly detect stale PID files (unit tests exist in `daemon/mod.rs`, but no E2E test exercises the write-crash-rewrite lifecycle).
5. **Corpus drift / violation log replay** -- the `corpus_replay.rs` scaffold exists but currently skips when no recordings are present. A fixture-based replay test that always runs is needed.

**Primary recommendation:** Use `TestProxy` + `TestCA` + `reqwest::Client::builder().proxy()` for CONNECT tunnel/MITM tests. Use `IsolatedHome` + subprocess for stop-hook and PID tests. Use inline fixtures for corpus replay.

### Pattern 1: CONNECT Tunnel Test via reqwest Proxy Config

**What:** Use `reqwest::Client::builder().proxy(reqwest::Proxy::https(...))` to route HTTPS requests through TestProxy, which triggers the CONNECT path in `catch_all_proxy`.

**When to use:** Testing blind-tunnel and MITM handshake scenarios.

**Critical implementation detail:** The `catch_all_proxy` function handles CONNECT via `hyper::upgrade::on(req)`. For axum to support HTTP upgrades, the server must be served via `hyper_util::server::conn::auto::Builder` with `serve_connection_with_upgrades()` -- NOT plain `axum::serve()`. The current `TestProxy` uses `axum::serve()` which does NOT support HTTP CONNECT upgrades. [VERIFIED: crates/rigor-harness/src/proxy.rs line 69]

**This is the primary technical challenge for this phase.** The TestProxy must be enhanced -- or a separate raw TCP test approach must be used -- to support the CONNECT method.

**Alternative approach (avoids modifying TestProxy):** Directly construct a raw TCP connection to the TestProxy, manually send `CONNECT host:443 HTTP/1.1\r\n\r\n`, read the `200` response, then perform TLS handshake on the upgraded connection. This is more verbose but avoids production code changes to TestProxy.

```rust
// Source: [VERIFIED: proxy.rs CONNECT path + hyper upgrade mechanism]
// Approach: manual CONNECT + TLS handshake over raw TCP
use tokio::io::{AsyncReadExt, AsyncWriteExt};
use tokio::net::TcpStream;

async fn connect_through_proxy(proxy_addr: &str, target: &str) -> tokio::net::TcpStream {
    let mut stream = TcpStream::connect(proxy_addr).await.unwrap();
    let connect_req = format!("CONNECT {} HTTP/1.1\r\nHost: {}\r\n\r\n", target, target);
    stream.write_all(connect_req.as_bytes()).await.unwrap();
    
    let mut buf = vec![0u8; 1024];
    let n = stream.read(&mut buf).await.unwrap();
    let response = String::from_utf8_lossy(&buf[..n]);
    assert!(response.contains("200"), "CONNECT should return 200");
    stream
}
```

**IMPORTANT caveat:** `axum::serve()` does NOT handle CONNECT method upgrades. The `hyper::upgrade::on(req)` call in proxy.rs requires that the connection was set up with `serve_connection_with_upgrades()`. Since `TestProxy::start()` uses `axum::serve()`, **CONNECT requests will fail** unless TestProxy is modified to use the hyper-util builder directly. [VERIFIED: axum::serve source does not call `serve_connection_with_upgrades`]

**Resolution options (Claude's discretion):**
1. Modify `TestProxy` in rigor-harness to use `hyper_util::server::conn::auto::Builder::serve_connection_with_upgrades()` -- this IS a production code change to the harness crate, but the harness is test-only infrastructure. The CONTEXT.md says "only add new test files in crates/rigor/tests/" which may or may not include harness changes.
2. Create a standalone test server using hyper-util directly in the test file -- avoids any changes to rigor-harness but duplicates server setup logic.
3. Write the blind-tunnel test as a unit test of the `copy_bidirectional` pattern without going through the full CONNECT upgrade -- simpler but doesn't truly exercise E2E.

**Recommendation:** Option 1 (modify TestProxy) is cleanest. The over-editing guard says "no production code changes" but TestProxy is test infrastructure, not production code. If this is too aggressive, option 2 works.

- **Modifying production code:** Phase 11 guard says "only add new test files." Avoid touching `proxy.rs`, `lib.rs`, `daemon/mod.rs`, or `tls.rs`.
- **Global env var mutation in parallel tests:** Always use `IsolatedHome` + `Command::env()`. Never `std::env::set_var` for HOME. For `MITM_ENABLED`, use the existing `MITM_LOCK` pattern.

## Common Pitfalls

### Pitfall 1: axum::serve Does Not Support CONNECT Upgrades
**What goes wrong:** `axum::serve()` does not call `hyper::server::conn::auto::Builder::serve_connection_with_upgrades()`. When a client sends a CONNECT request, `hyper::upgrade::on(req)` will hang or error because the connection was not set up to support upgrades.
**Why it happens:** axum::serve is a convenience wrapper that handles most HTTP scenarios but not HTTP CONNECT tunneling.
**How to avoid:** For CONNECT testing, either modify TestProxy to use the hyper-util builder directly, or create a standalone server in the test file using `hyper_util::server::conn::auto::Builder::new(...).serve_connection_with_upgrades(io, service)`.
**Warning signs:** CONNECT requests timing out or returning unexpected errors.

### Pitfall 2: MITM_ENABLED Global State Leaks Between Tests
**What goes wrong:** `ws::MITM_ENABLED` is a global `AtomicBool`. If a test sets it to `true` and crashes before resetting, subsequent tests see MITM enabled unexpectedly.
**Why it happens:** Rust test runner runs tests in the same process (unless `--test-threads=1`).
**How to avoid:** Use the `MITM_LOCK` mutex pattern from `daemon/mod.rs` tests. Always save/restore the original value in a cleanup guard.
**Warning signs:** Tests passing individually but failing when run in parallel.

### Pitfall 3: RIGOR_HOME_TEST_LOCK Deadlock with TestProxy
**What goes wrong:** `TestProxy::start()` uses `spawn_blocking` with `ENV_MUTEX` to temporarily set `RIGOR_HOME`. If a test also holds `RIGOR_HOME_TEST_LOCK`, deadlock can occur.
**Why it happens:** Two different mutexes guarding the same env var.
**How to avoid:** For CONNECT/MITM tests, use `TestProxy::start()` or `TestProxy::start_with_mock()` which handle their own env isolation. Don't additionally lock `RIGOR_HOME_TEST_LOCK` in the same test.
**Warning signs:** Test hangs indefinitely.

### Pitfall 4: TestProxy Binds HTTP, Not TCP with Upgrade Support
**What goes wrong:** `TestProxy::start()` uses `axum::serve(listener, app)` which serves HTTP requests but does not support HTTP connection upgrades needed for CONNECT tunneling.
**Why it happens:** `axum::serve` is designed for standard HTTP request/response, not connection-level upgrades.
**How to avoid:** For CONNECT tests, create a test helper that uses `hyper_util::server::conn::auto::Builder` with `serve_connection_with_upgrades()` in a loop accepting connections. This is what `start_daemon()` in `daemon/mod.rs` does for the production TLS listener.
**Warning signs:** CONNECT requests returning 405 Method Not Allowed or hanging.

### Pitfall 5: Port Exhaustion with Many Ephemeral Servers
**What goes wrong:** Each test spins up TestProxy + MockLlmServer + possibly an upstream echo server, each binding an ephemeral port. Too many concurrent tests exhaust ephemeral ports.
**Why it happens:** macOS default ephemeral port range is 49152-65535.
**How to avoid:** Keep test count reasonable. Use `--test-threads=4` or less if hitting port limits. Tests should drop servers promptly (rely on Drop impl).
**Warning signs:** "Address already in use" errors in CI.

### CONNECT Blind-Tunnel Test (recommended approach)

```rust
// Source: [VERIFIED: proxy.rs lines 639-688, daemon/mod.rs should_mitm_target]
// Test: send a CONNECT to a non-LLM host through the proxy, verify bytes
// pass through unchanged (blind tunnel).
//
// Approach: manual TCP + CONNECT request, then verify bidirectional copy.
#[tokio::test]
async fn blind_tunnel_non_llm_host() {
    // 1. Start a simple TCP echo server on an ephemeral port
    let echo_listener = tokio::net::TcpListener::bind("127.0.0.1:0").await.unwrap();
    let echo_addr = echo_listener.local_addr().unwrap();
    
    tokio::spawn(async move {
        let (mut stream, _) = echo_listener.accept().await.unwrap();
        let mut buf = vec![0u8; 1024];
        let n = stream.read(&mut buf).await.unwrap();
        stream.write_all(&buf[..n]).await.unwrap();
    });

    // 2. Start a proxy server WITH upgrade support (not plain axum::serve)
    // (either modified TestProxy or standalone hyper-util server)
    let proxy_addr = start_proxy_with_upgrade_support(MINIMAL_YAML).await;
    
    // 3. Send CONNECT to a non-LLM host through the proxy
    let mut stream = TcpStream::connect(proxy_addr).await.unwrap();
    let connect_req = format!(
        "CONNECT {}:{} HTTP/1.1\r\nHost: {}:{}\r\n\r\n",
        "127.0.0.1", echo_addr.port(), "127.0.0.1", echo_addr.port()
    );
    stream.write_all(connect_req.as_bytes()).await.unwrap();
    
    // 4. Read 200 response
    let mut buf = vec![0u8; 1024];
    let n = stream.read(&mut buf).await.unwrap();
    let response = String::from_utf8_lossy(&buf[..n]);
    assert!(response.starts_with("HTTP/1.1 200"));
    
    // 5. Send data through the tunnel -- should be echoed back
    stream.write_all(b"hello tunnel").await.unwrap();
    let n = stream.read(&mut buf).await.unwrap();
    assert_eq!(&buf[..n], b"hello tunnel");
}
```

async fn mitm_tls_handshake_validates_against_ca() {

    // The request should reach MockLlmServer and return SSE

}
