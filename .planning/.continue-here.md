## Critical Anti-Patterns (severity: blocking)

### 1. ENV_LOCK acquisition order in tests
**What it is:** Tests using `TestProxy` that mutate env vars (e.g. `RIGOR_NO_RETRY`) MUST acquire `rigor_harness::env_lock::ENV_LOCK` AFTER calling `TestProxy::start_with_mock()`, never before.

**How it manifested:** During the brutal-critic remediation, the C2+C6 unification put a single ENV_LOCK in rigor-harness. Tests that pre-acquired the lock then called `TestProxy::start_with_mock()` deadlocked because TestProxy itself acquires the same non-reentrant `std::sync::Mutex` on a `spawn_blocking` thread. A short-lived `try_lock` workaround broke serialization (b3_pii_redact tests received wrong mock URLs from racing tests). The correct fix is the documented order: TestProxy first, ENV_LOCK after.

**Structural prevention:** All four affected test files (`b1_kill_switch.rs`, `b2_auto_retry.rs`, `proxy_pipeline_cohesion.rs`, `pid_lifecycle_e2e.rs`) now follow this order. The TestProxy doc comment in `crates/rigor-harness/src/proxy.rs` notes the contract. Future env-mutating tests must mirror the b1/b2 pattern.
